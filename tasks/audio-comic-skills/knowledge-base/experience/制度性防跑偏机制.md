# 制度性防跑偏机制

## 核心问题

AI 跑偏的本质不是"不会"，而是三个作弊行为：

| 失败模式 | 表现 |
|---------|------|
| **假完成** | 模型觉得"做完了"，但只是做了个样子 |
| **绕验证** | 偷懒绕过验证，或验证本身不可复现 |
| **跑偏自信** | 跑偏了还自信满满，人接手时一地鸡毛 |

## 核心解法：职责正交化

**写的不验，验的不写。** 不是靠模型更聪明来防跑偏，而是通过制度设计让模型"即使想跑偏也跑偏不了"。

```
Supervisor（主编 Agent）     Worker（执行 Agent）
─────────────────────       ─────────────────────
派发任务                      读取 Startup Ritual
执行验收（运行脚本/截图）       实现功能 + 写 BUNDLE 行
写 EVIDENCE（PASS/FAIL）      制作 Validation Bundle
更新 passes 状态              ❌ 禁止声明 PASS/FAIL
创建 Git 快照                 ❌ 禁止修改 passes 状态
写 progress.txt 交接日志      ❌ 禁止创建 git 提交
```

## 制度设计四要素

### 1. 角色禁止边界（Governance）

明确每个 Agent 能做什么、**绝对不能做什么**。越权操作本身即视为违规。

### 2. 不可篡改的证据链

每个任务完成后交付 **Immutable Run Folder**：
- 命名含时间戳，永不覆盖历史
- Supervisor 验证 PASS 后才写 EVIDENCE
- 证据链断裂 = 任务未完成（HARD GATE）

### 3. 两层账本

- **过程账本（tasks.md 替代品）**：记录重试、报错、试错细节，人机共见
- **结果账本（feature_list.json 替代品）**：只有 Supervisor 在完整证据链成立后翻转 passes=false → true

两层账本通过环节 ID 刚性绑定。模型可能"看起来完成了过程账本"，但结果账本的 passes=false 暴露它。

### 4. Startup Ritual（启动仪式）

每个 Worker 启动前必须读取：
1. 上一环节的 `handoff_log.txt`（历史交接记录）
2. `product_acceptance.json`（当前产品完成度）
3. archive/ 最近 Run 的产出摘要
4. 当前 commit hash（Git 管理时）

写入 `logs/startup.txt` 证明"我看过以前发生什么了"。

## HARD GATE（硬门槛）

任务标记为 DONE 必须同时满足：
- 有 `EVIDENCE (RUN #n)` 行（Supervisor 写）
- 有 `VALIDATION_BUNDLE: ...` 指向的产物目录存在
- 有 `WORKER_STARTUP_LOG: ...` 证明 Worker 读过历史
- 有 `RESULT: PASS`
- 有 Git Commit + Diffstat/Files
- 至少一条证据（截图/音频样本/视频片段）
